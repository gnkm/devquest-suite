# コードレビュー チェックリスト

> **目的**: 一貫した観点で PR をレビューし、品質と学習効果を最大化する。必要に応じて項目を組織方針へ合わせて調整してください。

---

## 0. レビュー準備
- [ ] CI がグリーン（ビルド・テスト・Lint 通過）
- [ ] PR 説明が要点を押さえている（背景・変更点・影響範囲）
- [ ] 変更量が適切（巨大 PR は分割を依頼）

---

## 1. 設計・アーキテクチャ
- [ ] 仕様／タスク README と整合している
- [ ] 単一責任の原則を満たす
- [ ] 依存関係が適切（循環依存なし・DI/インターフェース活用）
- [ ] 将来の拡張・保守性を考慮した構造になっている

---

## 2. 実装・ロジック
- [ ] アルゴリズムやデータ構造が妥当
- [ ] エッジケースを考慮（null／空配列／境界値など）
- [ ] エラーハンドリングが適切（例外種類・ログレベル）
- [ ] 並行／非同期処理で競合やデッドロックがない

---

## 3. テスト
- [ ] 新規／変更ロジックに対するユニットテスト追加
- [ ] 主要パス＋失敗パスをカバー
- [ ] カバレッジ 80% 以上を維持
- [ ] テストが決 determin istic（ランダム／外部依存を避ける）

---

## 4. ドキュメンテーション
- [ ] クラス・関数に Docstring／コメントがある
- [ ] README／設計資料に影響があれば更新済み
- [ ] コードの意図がコメントで補足されている（“なぜ” が明確）

---

## 5. コーディングスタイル
- [ ] Linter／Formatter 準拠（ruff）
- [ ] 意味のある命名（英語・一貫性）
- [ ] マジックナンバー排除、定数化
- [ ] ファイル／関数の長さが過度に長くない

---

## 6. パフォーマンス・リソース
- [ ] 不要な O(n²)・I/O ブロッキングがない
- [ ] メモリリークや過大メモリ確保の懸念なし
- [ ] キャッシュや遅延評価の活用が適切

---

## 7. セキュリティ & ライセンス
- [ ] インジェクション・XSS・CSRF 等の脆弱性チェック
- [ ] 機密情報（キー・パスワード）がハードコードされていない
- [ ] 依存ライブラリのライセンスを確認

---

## 8. 生成 AI 利用
- [ ] AI 生成コードの場合、ソース／ライセンス確認メモが PR に添付
- [ ] プロンプトや出力に機密情報が含まれていない

---

## 9. レビュー後プロセス
- [ ] すべてのコメントに対する対応済み・未対応が明示
- [ ] 追加コミットは Squash か Rebase で履歴を整理
- [ ] Reviewer と Author 双方が "LGTM" を確認後マージ

---
